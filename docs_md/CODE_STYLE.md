# 代码编写与可读性约定

这份文档记录当前项目在新任务下的代码规则，重点是可读性、实验复现性和约束显式化。

## 1. 文件头说明要求

每个新建或明显重构的代码文件都应在文件开头写简要说明注释。

对于 Python 文件，建议使用模块级 docstring，至少写清 4 件事：

- 这个文件是做什么的
- 主要输入是什么
- 主要输出是什么
- 适用于哪类任务约束

## 2. 新任务约束必须显式写出

如果脚本服务于当前主线，应明确说明：

- query 是任意单张无人机影像
- query 不带地理信息
- query 不保证为正射影像
- 是否依赖任何分辨率处理或几何假设

## 3. 历史逻辑隔离

- 不要直接复用旧任务脚本而不做说明
- 如确需参考 `old/` 中脚本，应在新脚本头部明确写出继承关系和修改点
- 不要让新脚本的命名继续绑定旧的 `200m same-scale` 口径

## 4. 可读性要求

- 一个脚本尽量只负责一类清晰任务
- 关键假设、阈值、输入前提应显式写出
- 对影响检索正确性的核心逻辑补充简短注释
- 输出文件命名应体现任务和方法，而不是历史阶段编号

## 5. 结果输出要求

如果后续脚本产生实验结果，建议至少输出：

- 配置说明
- 每 query 结果明细
- 汇总指标
- 失败样例入口

并同步更新 `docs_md/RESULTS_INDEX.md` 与 `docs_md/PROJECT_PROGRESS.md`。
