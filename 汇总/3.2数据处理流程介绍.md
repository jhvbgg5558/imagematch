# 3.2 数据处理流程介绍

## 3.2.1 流程概述

当前 DOM+DSM 实验的数据处理流程覆盖从原始 DOM/DSM 数据预处理、候选库构建、图像检索、几何重排、DSM 三维采样、PnP 位姿恢复，到多层定位精度检验的完整链路。该流程的核心原则是 runtime 阶段不使用 query 地理真值，所有 query truth、AT reference 和 DOM truth 只在离线验证阶段使用。

当前正式说明以 CaiWangCun DOM/DSM full-replacement full run 为主。实验根目录为：

`D:\aiproject\imagematch\new3output\nadir_009010_caiwangcun_domdsm_fullreplace_full_2026-04-21\`

对应 gate 根目录为：

`D:\aiproject\imagematch\new3output\nadir_009010_caiwangcun_domdsm_fullreplace_gate_2026-04-20\`

full run 沿用 gate 已验证的算法口径和参数，但扩展到 `40/40` 个 009/010 nadir query。数据处理上不覆盖 gate 目录，而是在独立 full root 中复制、重写、校验或重建所有关键资产。

## 3.2.2 输入数据

### 3.2.2.1 UAV query

query 使用 009/010 两条 UAV 航线中的 nadir 图像，每条航线选择 20 张，共 40 张，query ID 为 `q_001` 至 `q_040`。query 图像在 runtime 检索和位姿求解中被视为不带地理信息的单张 UAV 图像。即使项目中存在 query truth seed 或 AT reference，它们也只用于离线验证、报告和异常分析，不参与候选检索、重排或 PnP 求解。

full run 复用 gate 已验证的 query-side 资产：

- `selected_queries`
- `query_inputs`
- `query_features`
- `query_truth`

其中 `query_features` 用于 DINOv2 检索，`query_truth` 只用于 offline validation。

### 3.2.2.2 CaiWangCun DOM/DSM

CaiWangCun 数据包括成品 `0.14m DOM` 和 `0.14m DSM` 切片。早期原始路径为：

`D:\数据\武汉影像\CaiWangCun-DOM`

后续如需重新从原始切片生成 mosaic，应改查：

`E:\数据\武汉影像`

当前 full run 不依赖重新读取外部原始切片，而是使用实验 root 内已经物化的 branch-local mosaic。

full run 内的 DOM/DSM mosaic 路径为：

- `source_mosaic\caiwangcun_ortho_0p14m_epsg32650.tif`
- `source_mosaic\caiwangcun_dsm_0p14m_epsg32650.tif`
- `source_mosaic\caiwangcun_mosaic_summary.json`

DOM 为 RGB `uint8`，DSM 为单波段 `float32`，统一坐标系为 `EPSG:32650`，分辨率约 `0.14m`。

### 3.2.2.3 模型与运行资产

实验使用 DINOv2 进行全局图像特征检索，使用 FAISS 建立候选库索引，使用 RoMa v2 进行 query 与候选 DOM tile 的局部几何匹配和重排，使用 OpenCV PnP RANSAC 与 refinement 进行粗位姿恢复。

模型和算法参数在 gate 与 full run 中保持一致。实验重点不是改模型，而是验证 DOM/DSM 和候选库派生资产完整替换后，定位链路是否稳定。

## 3.2.3 DOM/DSM 数据预处理

### 3.2.3.1 切片拼接与坐标统一

首先读取 CaiWangCun DOM 与 DSM 切片，按 GeoTIFF 元数据处理源坐标系，并统一重投影到当前 formal 链路使用的 `EPSG:32650`。DOM 和 DSM 分别输出为统一 mosaic，并记录 mosaic summary。

输出包括：

- `source_mosaic\caiwangcun_ortho_0p14m_epsg32650.tif`
- `source_mosaic\caiwangcun_dsm_0p14m_epsg32650.tif`
- `source_mosaic\caiwangcun_mosaic_summary.json`
- `plan\caiwangcun_asset_manifest.csv`

该步骤确保 DOM 与 DSM 使用同一投影坐标系、同一覆盖参考和可比较的空间范围。前期实验表明，如果 DOM/DSM 或候选库资产来自不同框架，即使局部匹配质量较高，最终 predicted ortho 仍可能出现百米级偏移。

### 3.2.3.2 DOM/DSM 质量检查

预处理后需要检查：

- DOM 是否为 RGB 影像；
- DSM 是否为单波段 Float32；
- 分辨率是否约为 `0.14m`；
- DOM/DSM bounds 是否对齐；
- DSM nodata 是否可识别；
- CaiWangCun mosaic 是否覆盖 009/010 query 的主要活动区域。

这些检查结果写入 planning summary 和 asset manifest，作为后续 candidate library 和 DSM cache 的覆盖约束依据。

### 3.2.3.3 覆盖约束

候选 tile 必须落在 CaiWangCun DOM 和 DSM 的 fully-covered 交集内。超出覆盖范围或部分覆盖的 tile 不允许进入正式 candidate library，不允许回退到旧 satellite DOM、ODM LAZ 或 SRTM。

覆盖约束的目的不是扩大候选数量，而是保证进入定位链路的每一个候选都具备完整且同源的 DOM/DSM 支撑。当前 full-replacement candidate library 最终保留 `149` 个 tile。

## 3.2.4 Candidate library 构建

Candidate library 在 CaiWangCun DOM/DSM fully-covered 区域内生成。tile 尺度沿用 formal 实验口径：

- `200m`
- `300m`
- `500m`
- `700m`

输出包括：

- `candidate_library\tiles.csv`
- `candidate_library\tiles_native\`
- `candidate_library\roi_summary.json`

`tiles.csv` 是后续 candidate feature、FAISS mapping、retrieval、rerank、formal candidate manifest 和 DSM cache 的共同索引基础。每个 tile 记录其地理范围、尺度、图像路径和候选 ID。当前 full run 的 candidate tile 数为 `149`。

## 3.2.5 Candidate 特征与 FAISS 索引

对每个 CaiWangCun candidate tile 提取 DINOv2 pooler feature，并记录提取状态。输出包括：

- `candidate_features\caiwangcun_tile_dinov2_pooler.npz`
- `candidate_features\caiwangcun_tile_dinov2_status.csv`

然后基于 candidate feature 构建 FAISS inner-product index，并输出 mapping JSON：

- `faiss\caiwangcun_tiles_ip.index`
- `faiss\caiwangcun_tiles_ip_mapping.json`

full run 验收结果为 candidate tiles、candidate feature count 和 FAISS mapping count 均为 `149`，说明候选库、特征和索引数量一致。

## 3.2.6 DINOv2 粗检索

对 40 个 query 使用已有 query DINOv2 feature，在 CaiWangCun candidate FAISS index 中检索 Top-20 候选。输出包括：

- `romav2_rerank\coarse\retrieval_top20.csv`
- `romav2_rerank\coarse\summary_top20.json`
- `retrieval\retrieval_top20.csv`

full run 的 retrieval Top-20 输出为 `800` 行，对应 `40 * 20` 个 query-candidate pair。gate 阶段的粗检索指标为：

- `recall@1=0.675`
- `recall@5=0.95`
- `recall@10=0.975`
- `recall@20=0.975`
- `MRR=0.783571`

这些指标说明 CaiWangCun candidate library 能够较稳定地把真实区域召回到 Top-20 内，为后续 RoMa v2 重排和 PnP 求解提供候选基础。

## 3.2.7 RoMa v2 几何重排

DINOv2 粗检索得到的 Top-20 候选进入 RoMa v2 stage7 重排。RoMa v2 在 query 与 candidate DOM tile 之间建立局部几何匹配，输出每个 query-candidate pair 的匹配质量、内点支持和 rerank 排名。

输出包括：

- `romav2_rerank\stage7\*\reranked_top20.csv`
- exported pose retrieval：`retrieval\retrieval_top20.csv`

RoMa v2 的作用是把全局特征相似的候选进一步按局部几何一致性排序，使真正地理对应的候选更可能进入 best pose 求解结果。

## 3.2.8 Formal pose 输入构建

formal pose 阶段需要把 query、candidate、truth audit 字段、DSM cache 和 retrieval Top-20 组织成统一 manifest。关键输出包括：

- `pose_v1_formal\input\formal_query_manifest.csv`
- `pose_v1_formal\input\formal_candidate_manifest.csv`
- `pose_v1_formal\input\formal_truth_manifest.csv`
- `pose_v1_formal\input\formal_dsm_manifest.csv`
- `pose_v1_formal\manifest\pose_manifest.json`

其中 `formal_query_manifest.csv` 提供 query 图像路径、近似内参和运行所需元数据；`formal_candidate_manifest.csv` 绑定 candidate tile 图像和地理变换；`formal_dsm_manifest.csv` 绑定每个 candidate 对应的 DSM raster；`pose_manifest.json` 则是 formal pose runtime 的总入口。

## 3.2.9 DSM cache 构建

DSM cache 不从 ODM LAZ 栅格化，也不使用 SRTM。每个候选的 DSM raster 均从 full root 内的 CaiWangCun DSM mosaic 裁剪得到。

关键路径包括：

- source DSM：`pose_v1_formal\dsm_cache\source\caiwangcun_dsm_0p14m_epsg32650.tif`
- DSM rasters：`pose_v1_formal\dsm_cache\rasters\*.tif`
- DSM summary：`pose_v1_formal\dsm_cache\rasters\_summary.json`

full run DSM cache 结果为：

- `planned_count=119`
- `built_count=119`
- `failed_count=0`

这说明所有进入 formal pose 的 DSM 请求都成功从 CaiWangCun DSM mosaic 中裁剪生成，没有发生 fallback。

## 3.2.10 Pose runtime

formal pose runtime 对每个 query 的 Top-20 candidate 执行完整几何求解流程。

首先，RoMa matches 被导出为 query-candidate 匹配点。full run 的主要输出包括：

- `pose_v1_formal\matches\roma_matches.csv`

然后，系统把 candidate DOM 像素映射到投影坐标系，并从 DSM raster 中采样高程，生成 2D-3D 对应：

- `pose_v1_formal\sampling\sampled_correspondences.csv`

full run sampling 结果为：

接着，系统对每个 query-candidate pair 执行 PnP RANSAC 和 refinement：

- `pose_v1_formal\pnp\pnp_inliers.json`
- `pose_v1_formal\pnp\pnp_summary.json`

full run PnP 结果为：

最后，系统根据候选质量打分，为每个 query 选择 best pose：

- `pose_v1_formal\summary\pose_overall_summary.json`

## 3.2.11 定位精度检验流程

定位精度检验由 `pose_v1_formal\eval_pose_validation_suite_caiwangcun_truth\` 下的 validation suite 完成。该 suite 只用于 offline validation，不向 runtime 检索或 PnP 泄露 truth。

### 3.2.11.1 Layer-1：predicted ortho 与 DOM truth 对齐

Layer-1 首先为每个 query 构建 CaiWangCun DOM truth crop，再使用 best pose、query 图像和 DSM 渲染 predicted ortho。二者被放在同一 truth grid 上比较。

关键输出包括：

- `ortho_alignment\overall_ortho_accuracy.json`
- `ortho_alignment\viz_overlay_truth\`
- `ortho_alignment\viz_overlay_dom\`

full run Layer-1 结果为：

- evaluated `39/40`
- `center_offset_m mean=5.657901m`
- `ortho_iou mean=0.741115`
- `q_037` 为 `dsm_intersection_failed`

### 3.2.11.2 Layer-2：pose 与 AT/query reference 对比

Layer-2 将 best pose 与 AT/query reference pose 进行比较，检查相机中心位置和视线方向误差。

关键输出包括：

- `pose_vs_at\query_reference_pose_manifest.csv`
- `pose_vs_at\per_query_pose_vs_at.csv`
- `pose_vs_at\overall_pose_vs_at.json`

full run Layer-2 结果为：

- evaluated `40/40`
- `horizontal_error_m median=1.4296165m`
- `horizontal_error_m p90=3.652963m`
- mean `22.964179m` 被 `q_037` 的 `849.224956m` 异常值拉高

因此分析 full run 时不能只看 Layer-2 mean，还需要结合 median、p90、failure bucket 和 frame sanity。

### 3.2.11.3 Layer-3：tiepoint ground error

Layer-3 在 predicted ortho 与 truth ortho 之间提取同名点，并计算每个 RANSAC inlier 同名点的地面坐标差异。该层用于检查局部细节是否在地面平面上对齐。

关键输出包括：

- `tiepoint_ground_error\per_query_tiepoint_ground_error.csv`
- `tiepoint_ground_error\overall_tiepoint_ground_error.json`
- `tiepoint_ground_error\tiepoint_failure_buckets.csv`
- `tiepoint_ground_error\tiepoints\per_query_matches\<query_id>_tiepoints.csv`
- `tiepoint_ground_error\viz_tiepoints\`

每个成功 query 的 `<query_id>_tiepoints.csv` 字段为：

- `query_id`
- `match_index`
- `truth_col_px`
- `truth_row_px`
- `pred_col_px`
- `pred_row_px`
- `truth_x_m`
- `truth_y_m`
- `pred_x_m`
- `pred_y_m`
- `dx_m`
- `dy_m`
- `dxy_m`

差异定义为：

- `dx_m = pred_x_m - truth_x_m`
- `dy_m = pred_y_m - truth_y_m`
- `dxy_m = sqrt(dx_m^2 + dy_m^2)`

full run Layer-3 结果为：

- `tiepoint_eval_ok=39`
- `upstream_eval_failed=1`
- `tiepoint_xy_error_rmse_m=0.413562m`
- 生成 `39` 个 per-query tiepoint CSV
- `q_037` 因 upstream failed 不生成明细 CSV，并在 failure bucket 中记录

### 3.2.11.4 Frame sanity

Frame sanity 用于解释 predicted ortho 是否存在黑框、偏移、覆盖不足或坐标框架异常。它不是新的定位算法，而是对 validation 输出进行诊断分桶。

- `ortho_alignment\frame_sanity\overall_frame_sanity.json`
- `ortho_alignment\frame_sanity\failure_buckets.csv`
- `ortho_alignment\frame_sanity\figures\`

full run frame sanity 结果为：

- `ok_or_manual_review=39`
- `missing_inputs=1`
- `dsm_sample_valid_ratio_on_truth_grid mean=0.998752`
- `pred_valid_pixel_ratio mean=0.741114`
- usable 39-query set 的 `horizontal_error_m mean=1.778006m`

该结果说明大多数 query 的 DSM 支撑和 predicted ortho 覆盖均正常，`q_037` 是单独异常样本，不是全局 frame offset 复发。

## 3.2.14 流程复现注意事项

复现实验时最重要的是保持数据链路一致。不能把旧 candidate library、旧 retrieval/rerank、旧 DSM cache 或旧 truth grid 与新的 CaiWangCun DOM/DSM 混用。只替换 DOM 或 DSM 的半替换实验已经证明会导致 predicted ortho 倾斜、大面积黑框和 500m 级系统性偏移。

若重新生成 CaiWangCun mosaic，需要确认原始数据路径。当前原始数据可能已从 `D:\数据\武汉影像` 整体迁移到 `E:\数据\武汉影像`，但数据内容和目录结构不变。复现时应优先检查 `E:\数据\武汉影像`，并把生成后的 branch-local mosaic 固定写入实验 root，避免后续流程依赖外部路径变动。

最后，query truth、AT reference、DOM truth crop 和 tiepoint validation 都只能用于 offline validation。runtime 的候选检索、RoMa 重排和 PnP 求解必须保持 truth-free，这是该方法能否用于 GNSS 拒止视觉定位场景的基本前提。
